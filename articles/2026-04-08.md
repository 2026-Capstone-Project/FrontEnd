### 2. 반복 일정 동기화 식별자(Signature) 개선
반복 일정의 경우 동일한 `eventId`를 공유하기 때문에, 특정 날짜의 일정을 수정할 때 정확한 타겟팅이 필요합니다. **Yeonjin Kim** 개발자는 `useSyncEventTiming` 훅에서 `occurrenceDate`를 시그니처에 포함하여, 각 반복 인스턴스가 고유하게 식별되도록 수정했습니다.

```typescript
// src/shared/hooks/form/useSyncEventTiming.ts
// 기존 eventId 기반에서 occurrenceDate를 포함한 고유 시그니처로 변경
const signature = `${eventId}:${occurrenceDate ?? fallbackDate}:1:${start.getTime()}:${end.getTime()}`
if (lastSignatureRef.current === signature) return;
lastSignatureRef.current = signature;
```

### 3. Occurrence 기준의 데이터 업데이트 로직 정교화
일정의 범위를 업데이트할 때, 단순히 ID만 체크하는 것이 아니라 `occurrenceDate`와 `type`을 함께 검증하도록 로직을 강화했습니다. 이를 통해 반복 일정 중 특정일의 데이터만 정확히 변경되도록 보장합니다.

```typescript
// src/features/Calendar/utils/helpers/calendarPageHelpers.ts
export const updateEventRange = (
  events: CalendarEvent[],
  eventId: CalendarEvent['id'],
  start: Date,
  end: Date,
  occurrenceDate?: CalendarEvent['occurrenceDate'],
  type?: CalendarEvent['type'],
) =>
  events.map((item) =>
    item.id === eventId &&
    (type ? item.type === type : true) &&
    (occurrenceDate ? item.occurrenceDate === occurrenceDate : true) // 정확한 인스턴스 매칭
      ? { ...item, start: moment(start).toDate(), end: moment(end).toDate() }
      : item
  );
```

### 4. 일정-할일 전환 시 데이터 유지 기능 (Draft System)
사용자가 일정을 작성하다가 할일로 타입을 변경할 경우, 기존에 입력했던 제목, 설명, 색상 등의 데이터가 유실되지 않도록 `ItemEditorDraft` 타입을 도입하고 관련 훅들을 수정했습니다.

```typescript
// src/shared/types/modal/itemEditor.ts
export type ItemEditorDraft = {
  title: string;
  description: string;
  startDate: Date | null;
  endDate: Date | null;
  startTime?: string;
  endTime?: string;
  isAllday: boolean;
  eventColor: EventColorType;
  repeatConfig: RepeatConfigSchema;
  location: string;
  address: string | null;
}
```

이 변경사항은 `useScheduleEditorForm`과 `useTodoEditorForm`에 반영되어, 모달 내에서 타입이 스위칭될 때 `draftValues`를 통해 이전 입력 상태를 주입받습니다.

### 5. 캘린더 미리보기 및 폼 초기화 버그 수정
할일에서 일정으로 전환할 때 캘린더 상에 즉각적으로 변경된 타입이 반영되지 않던 문제를 `onEventTypeChange` 핸들러 추가를 통해 해결했습니다. 또한, `ItemEditorModal`에서 `endTime`을 계산할 때 불필요하게 복잡했던 조건문을 단순화하여 코드 가독성을 높였습니다.

```typescript
// src/shared/ui/Modals/ItemEditorModal/index.tsx
// Todo와 Schedule의 종료 시간 설정 로직 단일화
endTime: initialType === 'todo' ? formatTimeFromDate(baseStart) : formatTimeFromDate(baseEnd),
```

오늘의 작업으로 Calio 팀은 캘린더의 데이터 무결성을 한층 강화했으며, 사용자가 복잡한 반복 일정을 관리하거나 항목의 성격을 변경할 때 발생할 수 있는 혼란을 최소화했습니다.
